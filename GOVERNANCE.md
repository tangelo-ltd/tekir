## Decision Making

### Spec Changes

Changes to the TEKIR specification (`spec/`) follow this process:

1. **Propose** - Open a GitHub Issue with the `[Proposal]` tag
2. **Discuss** - Community feedback period (minimum 7 days for non-trivial changes)
3. **Refine** - Update the proposal based on feedback
4. **Merge** - Maintainers merge after consensus

Minor clarifications and typo fixes can go through regular PRs without a proposal.

### Code Changes

Changes to the reference implementation (`src/`) follow standard PR workflow:

1. Open a PR with a clear description
2. Maintainers review
3. Merge after approval

### Breaking Changes

Any change that would break existing TEKIR-compliant implementations requires:

1. A `[Proposal]` issue with a migration path
2. Minimum 14-day feedback period
3. A new spec version number

## Becoming a Maintainer

Active contributors who demonstrate understanding of the TEKIR spec and
consistent high-quality contributions may be invited as maintainers.

## Future Governance

As TEKIR grows, governance may evolve toward a more open model (e.g., a
technical steering committee). Any governance changes will follow the proposal
process above.
